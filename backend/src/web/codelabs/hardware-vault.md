## Welcome to the hardware vault

Understand the difference between "encrypted at rest" and "hardware-backed at rest", and
why the second one is what regulators actually want.

Any app can ship an encrypted SQLite database. Encryption alone proves nothing — if the
key lives on disk, a root user reads the key and decrypts. The Android Keystore moves the
key into the device's secure element (TEE / StrongBox). The key never leaves; encryption
happens *through* the Keystore via an opaque handle. Even with root, the cleartext key is
not recoverable.

> **Why this matters.** Token leaks are persistent-session compromises. A refresh token in
> plain SharedPreferences is one `adb pull` away from full account takeover. A refresh
> token sealed in AES-256-GCM with a Keystore key is useless to anyone but the device.

---

## Step 1: Decide what goes behind the vault

Not everything. The vault is for the smallest possible set of secrets:

- Refresh token (the persistent-session credential).
- Device-binding private key (already Keystore-resident as a key handle).
- Card PAN reveals (transient; never persisted, returned only on step-up).

Anything else — UI preferences, account balances, transaction caches — does not belong in
the vault. The vault is expensive (every read decrypts) and noisy in audits.

> **Why this matters.** Over-encrypting is a code smell. It hides what is actually
> sensitive and inflates the surface auditors have to review.

## Step 2: Generate the master AES key

A single AES-256-GCM key, alias `fortress.refreshVault.v1`. The `v1` suffix is a courtesy
for future rotation.

```kotlin
const val ALIAS_REFRESH = "fortress.refreshVault.v1"

fun ensureMasterKey() {
  val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
  if (ks.containsAlias(ALIAS_REFRESH)) return
  val spec = KeyGenParameterSpec.Builder(
    ALIAS_REFRESH,
    KeyProperties.PURPOSE_ENCRYPT or KeyProperties.PURPOSE_DECRYPT,
  )
    .setKeySize(256)
    .setBlockModes(KeyProperties.BLOCK_MODE_GCM)
    .setEncryptionPaddings(KeyProperties.ENCRYPTION_PADDING_NONE)
    .setRandomizedEncryptionRequired(true)
    .setIsStrongBoxBacked(true)
    .build()
  val kg = KeyGenerator.getInstance(KeyProperties.KEY_ALGORITHM_AES, "AndroidKeyStore")
  try {
    kg.init(spec)
  } catch (_: StrongBoxUnavailableException) {
    // Fallback: same spec without StrongBox. TEE still applies.
    kg.init(KeyGenParameterSpec.Builder(spec).setIsStrongBoxBacked(false).build())
  }
  kg.generateKey()
}
```

> **Why this matters.** `setRandomizedEncryptionRequired(true)` is what forces a fresh IV
> per call. GCM with a reused IV is broken; the OS enforces this for you only when you ask.

## Step 3: Seal a plaintext payload

Sealing produces a `(iv, ciphertext)` pair. Persist both — the IV is not secret but is
required to decrypt.

```kotlin
data class Sealed(val ivB64: String, val ciphertextB64: String)

fun seal(plaintext: ByteArray): Sealed {
  val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
  val key = ks.getKey(ALIAS_REFRESH, null) as SecretKey
  val cipher = Cipher.getInstance("AES/GCM/NoPadding").apply { init(Cipher.ENCRYPT_MODE, key) }
  val ct = cipher.doFinal(plaintext)
  return Sealed(
    ivB64 = Base64.encodeToString(cipher.iv, Base64.NO_WRAP),
    ciphertextB64 = Base64.encodeToString(ct, Base64.NO_WRAP),
  )
}
```

The IV is 12 bytes generated by the Keystore for you. Do not pass your own — that is how
people accidentally reuse IVs and turn GCM into a confidentiality-free cipher.

> **Why this matters.** GCM's security collapses on IV reuse. Letting the OS pick is the
> only safe path.

## Step 4: Open a sealed payload

Symmetric to Step 3. Decoding either piece returns the original bytes; failures throw a
typed exception so callers can decide what to do.

```kotlin
fun open(sealed: Sealed): ByteArray {
  val ks = KeyStore.getInstance("AndroidKeyStore").apply { load(null) }
  val key = ks.getKey(ALIAS_REFRESH, null) as SecretKey
  val iv = Base64.decode(sealed.ivB64, Base64.NO_WRAP)
  val cipher = Cipher.getInstance("AES/GCM/NoPadding")
  cipher.init(Cipher.DECRYPT_MODE, key, GCMParameterSpec(128, iv))
  return cipher.doFinal(Base64.decode(sealed.ciphertextB64, Base64.NO_WRAP))
}
```

> **Why this matters.** A decryption exception is information. The right reaction is not
> to silently wipe the vault but to surface the failure so on-call can investigate.

## Step 5: Wire up the TokenStore interface

A thin contract the rest of the app sees. Implementation backs onto the vault and
DataStore for the IV+ciphertext blob.

```kotlin
interface TokenStore {
  suspend fun saveRefresh(token: String)
  suspend fun loadRefresh(): String?
  suspend fun clear()
}

class KeystoreTokenStore(
  private val vault: KeystoreVault,
  private val store: DataStore<Preferences>,
) : TokenStore {

  override suspend fun saveRefresh(token: String) {
    val sealed = vault.seal(token.toByteArray(Charsets.UTF_8))
    store.edit { it[REFRESH_IV] = sealed.ivB64; it[REFRESH_CT] = sealed.ciphertextB64 }
  }

  override suspend fun loadRefresh(): String? {
    val prefs = store.data.first()
    val iv = prefs[REFRESH_IV] ?: return null
    val ct = prefs[REFRESH_CT] ?: return null
    return runCatching { String(vault.open(Sealed(iv, ct)), Charsets.UTF_8) }.getOrNull()
  }

  override suspend fun clear() {
    store.edit { it.remove(REFRESH_IV); it.remove(REFRESH_CT) }
  }
}
```

> **Why this matters.** Every other component in the app sees only `String?`. The vault
> and serialisation details stay encapsulated, which means swapping to StrongBox-only or
> bumping `v1 → v2` is a one-file change.

## Step 6: Bind the master key to user authentication

For high-stakes flows (transfer signing, PAN reveal) regenerate the *device-binding* key
with `setUserAuthenticationRequired(true)`. The refresh vault key stays without auth
requirements — otherwise the app cannot run before unlock.

```kotlin
fun generateDeviceBindingKey() {
  val spec = KeyGenParameterSpec.Builder(
    "fortress.deviceBinding.v1",
    KeyProperties.PURPOSE_SIGN or KeyProperties.PURPOSE_VERIFY,
  )
    .setAlgorithmParameterSpec(ECGenParameterSpec("secp256r1"))
    .setDigests(KeyProperties.DIGEST_SHA256)
    .setUserAuthenticationRequired(true)
    .setUserAuthenticationParameters(0, KeyProperties.AUTH_BIOMETRIC_STRONG)
    .setInvalidatedByBiometricEnrollment(true)
    .build()
  KeyPairGenerator.getInstance(KeyProperties.KEY_ALGORITHM_EC, "AndroidKeyStore")
    .apply { initialize(spec) }
    .generateKeyPair()
}
```

`setInvalidatedByBiometricEnrollment(true)` is the cheap mitigation for "attacker enrolled
their own fingerprint" — the key vanishes on enrolment change, forcing re-bind on the
server.

> **Why this matters.** A vault that cannot be opened until the user authenticates is a
> stronger claim than one that protects bytes only. The biometric prompt becomes part of
> the cryptographic boundary.

## Step 7: Probe StrongBox availability

StrongBox is a discrete secure element separate from the TEE. On supported devices (Pixel
3+, recent Samsung), it is the highest-assurance backing for Keystore keys. Not every
device has it; the spec must fall back gracefully.

```kotlin
fun isStrongBoxAvailable(context: Context): Boolean {
  return Build.VERSION.SDK_INT >= Build.VERSION_CODES.P &&
    context.packageManager.hasSystemFeature(PackageManager.FEATURE_STRONGBOX_KEYSTORE)
}
```

Surface the result on the Security Center screen so the user can see *which* hardware is
guarding their secrets. Transparency builds the kind of trust regulators reward.

> **Why this matters.** Pixel 3 launched in 2018. Eight years later, devices without
> StrongBox are still common. A vault that crashes on those devices is a vault that ships
> on no devices.

## Step 8: Rotate the master key

Rotation is straightforward when the alias is versioned. Generate `v2`, decrypt every
sealed payload with `v1`, re-seal with `v2`, delete `v1`.

```kotlin
suspend fun rotateMasterKey() {
  ensureMasterKey("fortress.refreshVault.v2")
  val prefs = store.data.first()
  val iv = prefs[REFRESH_IV] ?: return
  val ct = prefs[REFRESH_CT] ?: return
  val plaintext = vault.open(Sealed(iv, ct), "fortress.refreshVault.v1")
  val resealed = vault.seal(plaintext, "fortress.refreshVault.v2")
  store.edit { it[REFRESH_IV] = resealed.ivB64; it[REFRESH_CT] = resealed.ciphertextB64 }
  vault.deleteAlias("fortress.refreshVault.v1")
}
```

> **Why this matters.** Rotation has to be rare but possible. Hard-coding the alias makes
> rotation impossible without an app uninstall, which is operationally a non-starter.

## Step 9: Wipe on integrity loss

If the integrity verdict drops to `Untrusted`, wipe the vault. The legitimate user will
re-authenticate; the attacker on the rooted device loses the refresh token before they
can exfiltrate it.

```kotlin
suspend fun wipeOnHostileDevice() {
  if (integrityCheck.current() is IntegrityVerdict.Untrusted) {
    tokenStore.clear()
    sessionManager.signOut()
  }
}
```

> **Why this matters.** A vault that survives a compromised device is a vault that hands
> the attacker the same token the legitimate user has. Wipe is a recovery primitive, not a
> failure mode.

## Step 10: Test the unhappy paths

Force every branch:

1. Key vanished mid-session (factory reset + restore from cloud backup). `loadRefresh`
   should return `null`, app should land on Login.
2. Sealed bytes corrupted on disk. `open` throws; the runCatching path returns `null`; same
   recovery.
3. Biometric enrolment changed → device-binding key invalidated → server returns 401 on
   the next `/me/*` call → app re-runs enrolment.
4. StrongBox-only device that crashes on generation → fallback path kicks in, vault still
   works through the TEE.

> **Why this matters.** Recovery paths are the most-skipped, most-broken part of vault
> code. Test them once with Dev Mode toggles before a real user finds them.

## Wrap-Up

You can now seal and open arbitrary bytes through the Android Keystore with confidence:
AES-256-GCM, StrongBox where available, biometric-bound for the device-binding key,
clean rotation, and a sensible recovery story.
